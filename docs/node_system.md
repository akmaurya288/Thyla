# Thyla — Node System

> **Version:** 2.0.0  
> **Status:** Production Implementation Design  

---

## 1. Overview

The Node System defines the modular execution units of a Thyla graph. Each node strictly adheres to defined JSON schemas for robust typing while maintaining high execution efficiency.

Nodes execute completely state-free in themselves. Their inputs are hydrated instantly via the Orchestration Engine's localized context bindings, and outputs are returned entirely as fractional context patches.

---

## 2. Core Node Interface

Every backend Node implements a standard asynchronous trait. Output explicitly maps context update packages back to the engine.

```python
class NodeResult(BaseModel):
    status: Literal['completed', 'wait', 'failed']
    updates: dict[str, Any] = Field(default_factory=dict)
    wait_event_type: str | None = None
    error: str | None = None
```

```python
class ThulaNodeBase:
    id: str
    type: str

    async def execute(self, ctx: ExecutionContext) -> NodeResult:
        raise NotImplementedError
```
### 2.1 Execution Context
The context passed is isolated to the Node. The context contains lightweight getters/setters mapped logically around the Redis state bindings initialized by the parent Orchestrator. 
- Context contains `messages_window()` mapping to the restricted rolling window history.
- Context exposes `stream_flush()` to push streaming buffers.

## 3. Core Node Capabilities

### 3.1 LLM Node (Optimized Streaming)
The standard node for generating language behaviors via an LLM.

- **Buffering:** Direct token-by-token emission destroys network efficiency. The LLM Node pushes tokens to the localized `StreamingBuffer` attached to the `ExecutionContext`, which flushes based on grammar logic (punctuation) or batch timers (50ms).
- **Prompt Injection:** RAG variables are dynamically appended at the *end* of the prompt template sequence. This is done explicitly to ensure the system prompt prefix remains identical across executions, maximizing LLM provider prompt-caching hit rates.

### 3.2 APINode

Makes HTTP calls using safe blocks.
- Forces timeout bounds.
- Restricts resolution of private IPs to eliminate SSRF vulnerabilities at the network layer.

### 3.3 Function Node (Sanitized Subprocess)

Enables users to write arbitrary transformation blocks in Python or JS.
- **Simplification Rule**: No heavy WebAssembly runtimes or complex multi-container instantiations (e.g., gVisor) that create 3-second cold starts.
- **Execution Mechanism**: Standard ephemeral localized subprocesses mapped to basic read-only FS mounts, using default standard input/output bounds for JSON-RPC. Strict OS-level `ulimit` CPU and memory constraints enforce safety seamlessly.
- **Network Default**: Banned unless explicitly overridden.

### 3.4 Summary Node (Memory Controller)
A required node in looped/multi-turn conversational graphs.
- **Behavior:** Checks the orchestrator's current `conversation_turn` count. If turns > 10, the node requests a cheap extraction summary computation model (e.g., GPT-4o-mini), compresses the oldest 5 messages into a `historical_summary` string variable, and forces the framework to truncate the rolling window.

---

## 4. Failure and Branch Recovery

Schema outputs generated by nodes heavily dictate branch traversing.

If a Node throws an unhandled exception or times out:
1. The Orchestration engine traps the failure before the event loop crashes.
2. If `retry_policy` is active on the node configuration, temporal backoff triggers.
3. Upon hard-failure, execution routes implicitly to a defined `Fallback` node in the UI. If no fallback exists, the session cleanly transitions to `status = failed` and halts.
