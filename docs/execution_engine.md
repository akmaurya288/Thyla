# Thyla — Execution Engine

> **Version:** 2.0.0  
> **Status:** Production Implementation Design  

---

## 1. Dual Execution Model

The Orchestration Engine diverges execution into two explicit topologies to ensure WebRTC voice flows aren’t starved by backend webhooks, and asynchronous workflows don’t memory-leak API servers.

### 1.1 Real-Time Streaming Path (Streamer Nodes)
Triggered by: **WebSockets (Chat)** and **LiveKit (Voice/Video)**.

- **Execution Context:** Exists inside a dedicated `asyncio` task on the very API server terminating the client connection.
- **Why:** To stream generated token chunks (`text_delta`, `audio_delta`) natively to the active socket wrapper with sub-millisecond network hops.
- **Lifecycle:** 
  1. Connection opened. API node claims session ownership internally.
  2. Engine pulls LangGraph snapshot, executes iteratively.
  3. LLM tokens are buffered and flushed, piped instantly to TTS/Socket.
  4. Yields to event loop; awaits next socket message.

### 1.2 Async Background Path (Celery Workers)
Triggered by: **Webhooks**, **Email**, **WhatsApp**, **Cron events**.

- **Execution Context:** The API Gateway validates the payload, pushes it to `t:{tenant_id}:session_queue`, and returns HTTP 202. Celery workers pop the queue and perform traversal.
- **Why:** Reliability, retries, exponential backoff, scaling separately from internet-facing gateways.
- **Lifecycle:**
  1. Worker pulls message.
  2. Hydrates minimalist state from Redis.
  3. Steps through graph until graph yields `Wait` or concludes.
  4. Async external API calls are made safely.

## 2. Segmented State Management

To prevent I/O blocking from writing full multi-kilobyte graphs to Redis on *every node completion*, state is partitioned.

### 2.1 The State Triangle

1. **Routing Metadata (Redis String)**
   - **Key:** `t:{tenant_id}:sess:{session_id}:meta`
   - **Contents:** `status`, `current_node`, `wait_event_type`.
   - **Update pattern:** Entire payload overwritten on node completion (typically < 100 bytes).

2. **Node Variables (Redis Hash)**
   - **Key:** `t:{tenant_id}:sess:{session_id}:vars`
   - **Contents:** Key-value pairs generated by nodes (e.g., `extracted_name="Alice"`).
   - **Update pattern:** Node pushes only deltas using `HSET` operations.

3. **Conversation Memory (Postgres Table + Sliding Window)**
   - **Cold Storage:** Every message appended to `conversation_turns` (Postgres).
   - **Hot Memory Window:** The orchestrator maintains `last_k_messages` in Redis.
   - **Summarization:** When messages exceed limit (e.g., > 15 turns or token limit), the `Summarizer Task` compresses older context into a `summary_variable` and slides the window forward.

### 2.2 LangGraph Integration Strategy
Instead of using standard LangGraph `PostgresSaver` mapping giant chunks per step, implement a custom `BaseCheckpointSaver` that emits **JSON Patches** to Redis for high frequency tracking, and snapshots to Postgres only on `Wait` or `End` transitions.

## 3. Execution Lifecycle & Concurrency

### 3.1 Resolving 'Resume' Concurrency (The Inbox Model)
Race conditions arise when a user sends a message while the agent is midway through processing, or multiple webhooks fire for the same session. Instead of discarding events via locks (yielding HTTP 409), Thyla uses a unified inbox queue per session.

**Flow:**
1. Incoming trigger drops standard `UnifiedMessage` into `t:{tenant_id}:sess:{session_id}:inbox` (Redis List).
2. The orchestrator checks this inbox. If the session is `waiting`, it pops the message and transitions to `running`.
3. If the session is already `running`, the message stays in the queue. 
4. Upon graph completion/waiting step, the orchestrator checks the inbox before suspending. If messages exist, it loops.

### 3.2 Graph Traversal Boundaries
Execution relies on a topological BFS handling Condition routing:
```python
async def step_graph(session_id: str, inbox_message: UnifiedMessage):
    # 1. Load Hot State
    meta, vars = await load_hot_state(session_id)
    
    # 2. Assign Scope
    ctx = ExecutionContext(session_id=session_id, variables=vars, message=inbox_message)
    
    while True:
        node = get_node(meta.current_node)
        
        # 3. Execute
        result = await node.execute(ctx)
        
        # 4. Partial State Flush
        if result.updates:
            await redis.hset(f"{session_id}:vars", mapping=result.updates)
            
        if result.status == 'wait':
            # Snapshot cold state to Postgres before sleeping
            await save_cold_snapshot(session_id, ctx)
            meta.status = 'waiting'
            await redis.set(f"{session_id}:meta", meta)
            break
            
        meta.current_node = resolve_edges(node, result, ctx.variables)
```

## 4. Streaming Optimization (Token Batching)

Emitting chunk-by-chunk over pub/sub per LLM token degrades systems quickly. Thyla employs adaptive token batching.

**Rule Set:**
1. **Punctuation Triggers:** Flush buffer to client immediately if encountering `.`, `?`, `!`, or `\n`.
2. **Time/Size Limits:** Flush buffer if > 40ms have elapsed or > 10 tokens have accrued since last flush.
3. **Voice Fast-Path:** If channel is `voice`, flush strictly by punctuation (clause boundaries) to allow clean TTS chunk generation, rather than raw token spew.

```python
class StreamingBuffer:
    def __init__(self, flush_callback, is_voice: bool):
        self.buffer = ""
        self.flush_callback = flush_callback
        self.is_voice = is_voice
        self.last_flush = time.monotonic()

    async def push(self, token: str):
        self.buffer += token
        
        if self.is_voice and token in [".", "?", "!", "\n"]:
            await self.flush()
        elif not self.is_voice and (len(self.buffer) > 30 or (time.monotonic() - self.last_flush > 0.05)):
            await self.flush()

    async def flush(self):
        if not self.buffer: return
        await self.flush_callback(self.buffer)
        self.buffer = ""
        self.last_flush = time.monotonic()
```

## 5. Reliability via Sweeper Reconciliation

Relying on Redis TTL expiry keyspace notifications to clean up dead sessions is fragile.

**The Sweeper Daemon:**
A background Cron/Celery-beat task runs every 60 seconds:
1. Queries Postgres: `SELECT session_id FROM sessions WHERE status IN ('running', 'resuming') AND updated_at < NOW() - INTERVAL '5 minutes'`
2. Identifies zombies (API server crashed midway through execution).
3. Pushes recovery triggers to the Celery queue to resume the graph from the last clean checkpoint, or auto-transitions them to `failed` depending on retry policies.
